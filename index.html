How would you design a scalable system / data warehouse to handle millions of requests per day?
ArchitecturePattern
▼

A scalable data system is built in layers, each with a specific role. The standard approach in Azure is the Medallion Lakehouse architecture:

  • Ingestion Layer: ADF, Event Hubs, or Kafka pull data from diverse sources. The goal is to decouple sources from processing so either side can change independently.
  • Bronze (Raw) Layer: Data lands as-is in ADLS Gen2 — no transformation. This is your immutable source of truth and replay foundation.
  • Silver Layer: Data is cleaned, validated, deduplicated, and joined. Runs in Databricks or Synapse Spark, written in Delta format for ACID compliance.
  • Gold Layer: Business-aggregated, query-ready tables served via Synapse Dedicated Pool or Serverless SQL for BI tools like Power BI.
  • Orchestration: ADF or Synapse Pipelines trigger each stage with retry logic, dependency management, and monitoring built in.
  • Scaling: Databricks auto-scales clusters; Synapse DWUs scale on demand; ADLS is inherently elastic — pay for what you use.
2
What are the differences between horizontal and vertical scaling? When would you use each?
Fundamentals
▼

Vertical scaling (Scale Up) means adding more CPU, RAM, or disk to the same machine. Simpler — no code changes — but has a hard ceiling and creates a single point of failure.

Horizontal scaling (Scale Out) means adding more machines or nodes. Requires distributed systems thinking but offers virtually unlimited ceiling.

  • Use vertical scaling for relational DBs that are hard to shard, or for quick wins on legacy systems.
  • Use horizontal scaling for Spark jobs, Kafka brokers, ADLS workloads — anywhere data partitions well across nodes.

How would you design a cost-optimized batch pipeline in Azure for large-scale daily processing?
Cost Optimization
▼
  • Spot/low-priority Databricks clusters — up to 90% cheaper for non-time-critical batch jobs. Add retry policies since spot nodes can be evicted.
  • Pause Synapse Dedicated Pool when not in use — you only pay for DWUs when the pool is running. Use ADF schedules to auto-pause post-job.
  • Process only incremental data using watermarks or CDC — avoid full-table scans every night.
  • ADLS lifecycle policies to automatically tier cold data to Cool or Archive storage.
  • Compress output files using Parquet + Snappy to reduce storage and I/O costs.
  • Right-size clusters: profile job memory and CPU, pick the smallest cluster that fits.
  • Azure Reserved Instances for always-on workloads — commit 1-3 years for up to 40% savings.

How would you process incremental data without creating duplicates (Idempotency)?
Pattern
▼

An idempotent pipeline produces the same result whether it runs once or ten times. This is critical because pipelines fail and need to re-run.

  • Watermark table: Store the last processed timestamp or ID. Each run reads only rows newer than the watermark. After a successful load, update the watermark.
  • Delta Lake MERGE (UPSERT): Match on a unique business key — update if the row exists, insert if new. Naturally handles duplicates from re-runs.
  • Partition overwrite: Write new data to a date partition, overwrite that partition on re-run. Avoids duplicates by design when partitioned by ingestion date.
  • Dedup in PySpark: df.dropDuplicates(["id", "updated_at"]) before writing.

A pipeline completed successfully but produced incorrect data. How would you debug it?
Debugging
▼

This is the hardest class of bug — the system thinks everything succeeded. Work through these steps:

  1. Narrow down the layer. Check Bronze → Silver → Gold sequentially. At which layer does data first look wrong?
  2. Check transformation logic. Are joins on the wrong key? Is a GROUP BY missing a column? Are window functions using the right partition/order?
  3. Check data quality at source. Was the source data itself corrupt, late-arriving, or duplicated?
  4. Check watermark/filter logic. Did the incremental filter cut off valid records or include old ones twice?
  5. Check date/timezone issues. UTC vs local time bugs are extremely common.
  6. Enable row counts + checksums. Add audit columns at each layer — row_count, checksum, load_timestamp — so discrepancies become visible.
  7. Delta Lake time travel: SELECT * FROM table VERSION AS OF N to compare data before and after a specific run.

A data pipeline is failing intermittently in production. How would you debug and fix it?
Debugging
▼

Intermittent failures do not always reproduce. Think in categories:

  • Infrastructure/transient: Network timeouts, API rate limits, throttling from source. Fix: retry policies with exponential backoff.
  • Data-driven: A bad record arrives on some runs (null in a non-nullable field, unexpected schema). Fix: schema validation + dead-letter queues.
  • Resource contention: Cluster runs OOM on days when data volume spikes. Fix: monitor run sizes, auto-scale, or increase shuffle partitions dynamically.
  • Concurrency: Two pipeline runs overlap and conflict on a shared table. Fix: set pipeline concurrency to 1 or use table-level locks.
🛠
Tools: ADF Monitor → Pipeline Runs → Activity runs. Databricks Job Run history for Spark logs. Azure Monitor + Log Analytics for pattern analysis across runs.

1
Under what networking conditions must you use a Self-Hosted Integration Runtime (SHIR) instead of Azure IR?
Networking
▼

The Integration Runtime (IR) is the compute engine that moves data in ADF. Azure IR is Microsoft-managed, lives in the cloud, and works for cloud-to-cloud transfers. It requires sources reachable over the public internet.

Use Self-Hosted IR (SHIR) — a Windows agent installed on a machine in your own network — when:

  • Data source is on-premises (SQL Server, Oracle, SAP) behind a corporate firewall.
  • Source is in a private VNet with no public endpoint and no peering to ADF's managed VNet.
  • You need to copy data between two private networks.
  • Source requires a local driver (ODBC, Oracle client) installed on the IR machine.
✅
The SHIR machine initiates outbound connections to ADF's cloud service over port 443 — no inbound firewall rules required.

How do you implement a Watermark-based incremental load in ADF to avoid re-processing old data?
Pattern
▼

A watermark is a "last processed" marker — usually a timestamp or integer ID. Each run fetches only records newer than this marker.

  1. Create a watermark table in Azure SQL: CREATE TABLE watermark (table_name VARCHAR, last_value DATETIME)
  2. Lookup Activity — read the current watermark value for the target table.
  3. Lookup Activity — read the current MAX value in source: SELECT MAX(updated_at) FROM orders
  4. Copy Activity — source query uses both values: WHERE updated_at > old_wm AND updated_at <= new_max
  5. Stored Procedure Activity — after successful copy, update the watermark to the new max value.
⚠️
Capturing the new max before the copy starts prevents late-arriving records from being missed on subsequent runs.

What is the difference between Copy Activity and Data Flow in ADF?
Core Concept
▼

Copy Activity is a lightweight, high-speed data movement tool. It moves data from source to sink with minimal transformation — column renaming, basic type casting. It does not spin up a Spark cluster; it runs on the Integration Runtime directly. Very fast, very cheap. Use it for ingestion — landing raw files in ADLS.

Data Flow (Mapping Data Flow) is a visual, code-free transformation engine backed by Apache Spark. You can join, aggregate, pivot, derive columns, parse JSON, and split streams — all via a drag-and-drop UI. ADF compiles your Data Flow into Spark code and runs it on an auto-created cluster.

💸
Data Flow is significantly more expensive — it spins up Spark compute. Only use it when transformation complexity justifies it. For simple copy-and-land, always use Copy Activity.

How do you implement parameterization in ADF pipelines?
Best Practice
▼
  • Pipeline parameters: Defined at pipeline level. Passed when triggering via schedule trigger, event trigger, or Execute Pipeline activity.
  • Dataset parameters: Make dataset properties dynamic — table name, file path, container name. Reference with @dataset().paramName.
  • Linked Service parameters: Parameterize connection strings so one linked service handles multiple environments (dev/prod).
  • Global parameters: Set once at ADF level, available across all pipelines — useful for environment-level config like storage account names.
  • ForEach + parameterized pipeline: Build a config-driven framework — store table names in a control config, loop with ForEach, call a generic pipeline per table with the table name as a parameter.
5
Describe how error handling and retry mechanisms work in ADF.
Reliability
▼
  • Activity retry: Each activity has configurable retry count (default 0) and retry interval. ADF re-runs the failed activity N times before marking it failed. Use for transient failures like network blips and throttling.
  • On Failure path: Each activity has Success, Failure, and Completion output paths. Connect activities on the Failure path to trigger cleanup, send alerts (Web Activity to Teams webhook), or log errors to a SQL table.
  • Timeout: Set timeouts on long-running activities to prevent hanging pipelines from consuming resources indefinitely.
  • Fault tolerance in Copy Activity: Skip incompatible rows and log them to a fault file in ADLS instead of failing the entire copy. Configure under Copy Activity → Settings → Fault tolerance.

How would you securely migrate on-premises SQL Server data into ADLS Gen2?
SecurityMigration
▼
  1. Install SHIR on a machine in the on-prem network that can reach the SQL Server.
  2. Create a Linked Service in ADF pointing to on-prem SQL Server via the SHIR. Store the password in Azure Key Vault — never hardcode credentials.
  3. Use Managed Identity for ADF to authenticate to ADLS Gen2 — no storage account keys needed.
  4. Configure a Copy Activity with the on-prem SQL linked service as source and ADLS Gen2 as sink. Use Parquet format for storage efficiency.
  5. Ensure encryption in transit: SQL Server uses TLS, ADLS uses HTTPS, SHIR to ADF cloud uses TLS over port 443.
  6. Implement incremental load using watermarks or SQL Server Change Tracking to avoid moving the full dataset on every run.
  7. For high-security requirements, use ExpressRoute or VPN for private connectivity instead of public internet traversal.

1
When would you choose Hash distribution over Round-Robin or Replicated tables in a Synapse Dedicated Pool?
Distribution
▼

Synapse Dedicated Pool distributes data across 60 distributions (physical storage units). How data lands across those distributions determines query performance.

  • Hash distribution: Rows with the same hash column value always land on the same distribution. Use on large fact tables (millions+ rows) frequently joined on the same column (e.g., customer_id). When two hash-distributed tables are joined on the same column, no data movement is required — a massive performance gain.
  • Round-Robin: Rows distributed evenly across all 60 distributions sequentially. Very fast writes. Use for staging tables, temp tables, or tables without a clear join key. Bad for query performance — joins always require data movement.
  • Replicated: A full copy of the table on every compute node. Use only for small dimension tables (<2GB). Eliminates all shuffle on joins with fact tables. Avoid for large tables — 60x storage cost and slow writes on updates.
💡
Rule of thumb: large table + frequent joins → Hash on join key. Small dimension table → Replicate. Staging area → Round-Robin.

What is the Hierarchical Namespace (HNS) in ADLS Gen2 and how does it optimize big data operations?
Fundamentals
▼

Regular Azure Blob Storage is a flat namespace — "folders" are just prefixes in blob names. Operations like renaming or deleting a "folder" require enumerating and updating every blob with that prefix — O(n) operations at scale.

ADLS Gen2 with HNS enabled creates a true hierarchical file system like HDFS. Directories are first-class objects with their own metadata. This enables:

  • Atomic rename: Renaming a directory is a single metadata operation — O(1). Critical for Spark commit operations, which write to a temp location then rename to final.
  • Atomic delete: Delete an entire directory in one operation rather than looping through every file individually.
  • POSIX-compliant ACLs: Set read, write, and execute permissions at the directory level, not just file level.
  • Performance at scale: Dramatically faster directory-level operations when you have millions of files.
⚠️
Always enable HNS when creating storage accounts for data lake workloads. It cannot be enabled after account creation.

Explain Synapse Serverless SQL Pools versus Dedicated SQL Pools — when do you use each?
Architecture
▼

Serverless SQL Pool: No provisioned compute. Query files directly in ADLS (Parquet, CSV, JSON, Delta) using T-SQL with OPENROWSET or external tables. Pay per TB scanned. Great for ad-hoc exploration, data discovery, and building views for Power BI on top of the data lake. Limitation: no indexes or materialized aggregates — every query scans files.

Dedicated SQL Pool: You provision DWUs — dedicated compute and columnar storage (CCI). Supports indexes, statistics, materialized views, and workload management. Pay per DWU-hour. Best for repeated, complex BI queries with multi-user concurrency and SLA requirements.

📌
Common pattern: Serverless SQL for the Silver lake layer (ad-hoc, flexible exploration) | Dedicated Pool for the Gold layer (performance-critical BI dashboards).

What is the difference between PolyBase and COPY INTO in Synapse?
Data Loading
▼

Both load data from ADLS into a Dedicated Pool table, but they differ significantly:

PolyBase is the older approach. Requires creating an External Data Source, External File Format, and External Table as separate objects before loading. More verbose setup, supports fewer authentication methods. Still functional but is the legacy path.

COPY INTO is the modern, preferred approach. A single T-SQL statement handles everything. Supports more file formats (CSV, Parquet, ORC), more authentication methods (SAS, Managed Identity, Service Principal), and handles bad rows gracefully with an ERRORFILE option.

COPY INTO dbo.orders
FROM 'https://mystorageacct.dfs.core.windows.net/container/orders/*.parquet'
WITH (FILE_TYPE = 'PARQUET', CREDENTIAL = (IDENTITY = 'Managed Identity'));

What is workload management in Synapse and why is it important?
Performance
▼

Without workload management, a single heavy query can starve all other users of DWU resources. Synapse workload management has three components:

  • Workload groups: Define resource pools with min/max DWU percentages — e.g., ETL jobs get 50%, dashboards get 30%, ad-hoc gets 20%.
  • Workload classifiers: Route specific users, labels, or query patterns to a workload group — e.g., etl_user → ETL group; label 'dashboard' → Dashboard group.
  • Workload importance: HIGH/MEDIUM/LOW priority within the queue. High-importance queries run first when resources are contested.

How do you secure data in Azure Synapse Analytics?
Security
▼
  • Network: Deploy Synapse in a managed VNet. Use Private Endpoints to block public internet exposure. Apply Synapse Firewall rules to whitelist trusted IP ranges.
  • Authentication: Azure AD integration with AAD users and groups — avoid SQL logins where possible. Use Service Principals for automated pipelines.
  • Authorization: RBAC at the workspace level (Owner, Contributor, Reader). SQL-level roles. Column-Level Security and Row-Level Security for fine-grained access.
  • Dynamic Data Masking (DDM): Mask sensitive columns (SSN, credit card) for unauthorized users at query time. Data is not changed on disk — masking is applied at query time only.
  • Encryption: Data at rest encrypted with Azure-managed keys or Customer-Managed Keys (CMK) in Key Vault. All data in transit uses TLS.
  • Auditing: Enable Synapse auditing to write audit logs to a Storage Account or Log Analytics for compliance and forensic investigation.

1
A Databricks job is failing with Out-of-Memory (OOM) errors. How would you troubleshoot it?
Troubleshooting
▼
  1. Identify where OOM occurs: Driver OOM → you're collecting too much data to the driver (collect(), toPandas(), large broadcast). Executor OOM → individual tasks are too memory-heavy.
  2. Check Spark UI: Go to Stages → look for tasks with very large shuffle read/write or skewed task sizes.
  3. Fix data skew: Add salting for skewed joins, or enable AQE (spark.sql.adaptive.enabled=true).
  4. Increase shuffle partitions: Default is 200. For large data, increase to 400–1000 to reduce per-partition size.
  5. Upgrade cluster node type to increase executor memory, or increase spark.executor.memory in cluster config.
  6. Reduce data being processed: Push filters earlier. Use select() to drop unused columns before joins.
  7. Replace Python UDFs with native Spark functions — UDFs serialize data to a Python process, consuming extra memory.
  8. Avoid collect() on large DataFrames. Sample or aggregate first before pulling data to the driver.

How do you monitor and troubleshoot failed jobs in Azure Databricks?
Monitoring

  • Cluster logs: Check driver stdout/stderr for Python exceptions. Check event log for cluster-level issues (node eviction, OOM kill).

  • Alert on failure: In Databricks Jobs, set email or webhook alerts on failure. Integrate with Azure Monitor for SLA tracking.

What is the difference between schema inference and predefined schema in PySpark? Which is better in production?
Schema
▼

Schema inference means Spark reads a sample of the data and auto-detects column names and types. Convenient in development, but problematic in production:

  • Expensive: requires an extra scan of the data before processing begins.
  • Unreliable: inferSchema on CSV might read "123" as integer, but rows later with "123abc" fail or produce wrong types.
  • Non-deterministic: schema can silently change if source data changes, breaking downstream code without any obvious error.

Predefined schema (StructType) — you explicitly declare every column name and type before reading:

schema = StructType([
    StructField("order_id", IntegerType(), False),
    StructField("amount", DoubleType(), True),
    StructField("created_at", TimestampType(), True)
])
df = spark.read.schema(schema).json("path/to/data")
✅
Always use predefined schema in production. It's faster, predictable, and forces you to explicitly handle schema changes — which is exactly what you want in a controlled pipeline.

How would you build a reusable Python ETL framework in Databricks?
Architecture
▼
  • Config-driven: Store pipeline configs in JSON, YAML, or a control SQL table. Code reads config at runtime — add a new pipeline by adding a config row, not writing new code.
  • Separate concerns: Organize into readers/, transformers/, writers/, and utils/ (logging, audit, schema validation).
  • Base classes: Abstract base class for Reader, Transformer, Writer with standard read(), transform(df), write(df) interfaces.
  • Audit logging: Every pipeline run writes to an audit table — pipeline name, run ID, start/end time, rows read/written, status, error message.
  • Package as a wheel: Build as a Python package, publish to Databricks repos, install on clusters with %pip install.
  • Unit tests: Use pytest + chispa (PySpark DataFrame equality testing). Run in CI/CD pipeline (GitHub Actions / Azure DevOps).

How do you handle Schema Drift using ADF and Databricks?
Schema Management
▼

Schema drift = source adds/removes/renames columns unexpectedly, breaking your pipeline.

In ADF Data Flow: Enable "Allow schema drift" on the source transformation. ADF will pass through unknown columns. Use "Auto mapping" in the sink.

In Databricks:

  • Read with mergeSchema = True for Parquet — Spark merges schemas across files with different columns.
  • For Delta: use mergeSchema = true on write. Delta handles additive changes automatically; destructive changes (column removal) need explicit ALTER TABLE DROP COLUMN.
  • Add a validation step that compares the incoming schema to the expected schema and sends an alert before writing rather than silently dropping new columns.
📋
For Power BI downstream: use a view layer in Synapse or Databricks SQL that selects only the stable columns BI depends on — insulating reports from upstream schema changes.

1
How do you handle Data Skewness in a PySpark cluster? Explain Salting.
Performance
▼

Data skew occurs when data is not evenly distributed across partitions. One partition may have 10 million rows while others have 10,000 — that one task takes 100x longer and becomes the bottleneck of the entire stage.

  • Salting (manual): Append a random integer (0 to N-1) to the join key on the skewed side. Explode the small side N times with each salt value. Join on the salted composite key. Drop the salt column after joining. This distributes one hot key across N partitions.
  • AQE: Enable spark.sql.adaptive.enabled=true and spark.sql.adaptive.skewJoin.enabled=true. Spark 3.x automatically detects and splits skewed partitions at runtime — no code changes needed.
  • Broadcast join: If the smaller side is <~200MB, broadcast it to eliminate the shuffle entirely.
  • Repartition on a composite key that distributes more evenly before the join.
2
Explain the difference between Vectorized UDF (pandas_udf) and standard UDF. Which is more efficient?
Performance
▼

Standard UDF: A Python function registered to run row-by-row. Spark cannot optimize it — it's a black box. Each row is serialized from JVM → Python process → processed → serialized back. This overhead is enormous, and UDFs disable Catalyst optimizations like predicate pushdown.

Vectorized UDF (pandas_udf): Data is batched into Pandas Series using Apache Arrow for zero-copy serialization. Much faster — typically 3-10x compared to row-at-a-time UDFs.

from pyspark.sql.functions import pandas_udf
import pandas as pd

@pandas_udf("double")
def multiply_by_two(s: pd.Series) -> pd.Series:
    return s * 2

df.withColumn("result", multiply_by_two(col("amount")))
🏆
Performance ranking: Native Spark functions > Vectorized pandas_udf > Row-at-a-time UDF. Always prefer native functions — they're fully optimized by the Catalyst engine.

Explain partitioning and bucketing in Apache Spark and when to use each.
Storage
▼

Partitioning splits data into directories based on column values — e.g., partitionBy("year", "month") creates year=2024/month=01/ folders. Queries filtering on the partition column skip entire directories (partition pruning). Best for low-cardinality columns like date, region, or status. High-cardinality columns (user_id) → millions of tiny files (small files problem).

Bucketing hashes data into a fixed number of files (buckets) based on a column. Files are in one directory; the structure is internal metadata. Key benefit: when two tables are bucketed on the same column with the same bucket count, Spark can join them without any shuffle.

df.write.bucketBy(64, "customer_id").sortBy("customer_id")\
    .saveAsTable("orders_bucketed")

Partitioning is for read-time pruning (filter performance). Bucketing is for join performance (shuffle elimination). They can be combined.

What is the difference between cache() and persist()? When should you use DISK_ONLY?
Memory
▼

cache() is shorthand for persist(StorageLevel.MEMORY_AND_DISK). It tries to store the DataFrame in memory and spills to disk if memory is insufficient.

persist() is the full API — choose the storage level explicitly:

  • MEMORY_ONLY: Fastest. If it doesn't fit, partitions are dropped and recomputed on access — no disk fallback.
  • MEMORY_AND_DISK: Default of cache(). Spills excess partitions to disk instead of recomputing. Safe for most use cases.
  • DISK_ONLY: Never uses memory. Use when the DataFrame is too large to fit in memory AND you want to avoid OOM errors. Slower but prevents crashes for extremely large intermediate results.
  • MEMORY_AND_DISK_SER: Serializes objects (smaller footprint) at cost of CPU for ser/deser.
⚠️
Always call unpersist() when done to free resources. Never cache a DataFrame used only once — it's pointless overhead.

Explain Broadcast Hash Joins. What is the default threshold size in Spark?
Optimization
▼

In a regular join, Spark must shuffle both tables across the network so matching keys land on the same executor. In a Broadcast Hash Join, the smaller table is fully copied ("broadcast") to every executor — no shuffle needed.

Default auto-broadcast threshold: 10MB (spark.sql.autoBroadcastJoinThreshold = 10485760). If the smaller table is under 10MB, Spark automatically broadcasts it.

from pyspark.sql.functions import broadcast
df_large.join(broadcast(df_small), "key")
⚠️
Broadcasting a 500MB table to 100 executors = 50GB total memory consumed. Increase the threshold carefully. Use the explicit broadcast() hint when Spark's size estimate is off.

What is Adaptive Query Execution (AQE)? How does it help at runtime?
Optimization
▼

Traditional Spark optimization happens at query parse time based on statistics that may be stale or missing. AQE (Spark 3.0+) re-optimizes the query plan at runtime using actual shuffle statistics.

  • Dynamic coalescing of shuffle partitions: If data is small, AQE merges small partitions → fewer tasks, less overhead.
  • Dynamic switch to broadcast join: If at runtime one side turns out small enough to broadcast, AQE switches from sort-merge to broadcast hash join mid-query.
  • Dynamic skew join handling: AQE detects skewed partitions (5x larger than median) and splits them into parallel sub-tasks.

Enable: spark.conf.set("spark.sql.adaptive.enabled", "true"). In Databricks, AQE is enabled by default from DBR 7.3+.

What is Spark Spill? How do you identify it in the Spark UI and resolve it?
Performance
▼

Spill occurs when a Spark task's in-memory data exceeds available executor memory. Spark writes excess to disk and reads it back when needed — disk I/O is 100-1000x slower than RAM.

Identify: Spark UI → Stages tab → look for non-zero Shuffle Spill (Memory) and Shuffle Spill (Disk) columns. Also check Stage detail → Task metrics for high "Disk bytes spilled".

  • Increase executor memory or upgrade node type.
  • Increase spark.sql.shuffle.partitions so each partition is smaller.
  • Fix data skew with salting or AQE — skewed tasks spill because they receive disproportionate amounts of data.
  • Push filters earlier to reduce data volume before expensive shuffle operations.

How do you handle the Small File Problem in a Data Lake? Coalesce vs. repartition?
Performance
▼

Thousands of tiny files require a metadata call per file and Spark creates one task per file — 10,000 files = 10,000 tasks = scheduler and storage service overwhelm.

  • coalesce(N): Reduces partition count without a full shuffle. Very efficient. Output may be slightly uneven. Use when reducing files before writing and balanced output is not critical.
  • repartition(N): Full shuffle — redistributes data evenly across N partitions. More expensive but produces balanced, evenly-sized output files.
df.coalesce(10).write.parquet("output/")      # cheap, slightly uneven
df.repartition(10).write.parquet("output/")   # expensive, perfectly even
💡
In Delta Lake: use the OPTIMIZE command which compacts small files into larger ones (default ~1GB) via bin-packing. Schedule nightly alongside Auto-Compaction for real-time mitigation.

When should Checkpointing be used in Spark? What happens if the checkpoint directory is deleted in Structured Streaming?
Streaming
▼

Batch checkpointing: For very long lineage chains (>~20 chained transformations), rdd.checkpoint() materializes the RDD to ADLS/HDFS, breaking the lineage so recomputations start from the checkpoint.

Structured Streaming checkpointing: The checkpoint directory stores the streaming state — offset tracking (which Kafka messages have been processed), aggregation state (running totals, windowed counts), and query progress metadata. It is how exactly-once processing is guaranteed.

🚨
If the checkpoint directory is deleted: The streaming query loses all state. On restart it begins fresh — it may re-read old data, aggregations reset to zero, and exactly-once guarantees are broken. For stateful aggregations this constitutes data loss. Never delete the checkpoint directory of a running or paused streaming query.
df.writeStream \
  .format("delta") \
  .option("checkpointLocation", "/mnt/checkpoints/mystream") \
  .start("/mnt/delta/output")

Write PySpark code to process and flatten nested JSON data.
Code
▼

Key functions: explode() for arrays (drops null rows), explode_outer() keeps null rows, col("struct.field") accesses nested struct fields.

from pyspark.sql.functions import col, explode_outer

# Read nested JSON
df = spark.read.option("multiline", "true").json("path/to/nested.json")

# Step 1: Explode the orders array (keep rows with null arrays)
df_orders = df.select(
    col("user_id"),
    col("name"),
    explode_outer(col("orders")).alias("order")
)

# Step 2: Access struct fields and explode nested items array
df_items = df_orders.select(
    col("user_id"),
    col("name"),
    col("order.order_id"),
    explode_outer(col("order.items")).alias("item")
)

# Step 3: Flatten struct fields into top-level columns
df_flat = df_items.select(
    col("user_id"),
    col("name"),
    col("order_id"),
    col("item.product"),
    col("item.qty")
)
df_flat.show()

1
How do you implement Row-Level Deletions in Delta Lake without rewriting the entire partition?
Delta Operations
▼

Traditional Parquet is immutable — deleting a row requires rewriting the entire file. Delta Lake introduces row-level deletes via a two-step mechanism:

from delta.tables import DeltaTable

dt = DeltaTable.forPath(spark, "/mnt/delta/orders")
dt.delete("order_id = 12345")

# Or with SQL:
spark.sql("DELETE FROM orders WHERE order_id = 12345")

How Delta handles it internally: Delta writes a new Parquet file with the non-deleted rows and marks the old file as deleted in the transaction log. The old file is logically gone but physically stays until VACUUM removes it after the retention period (default 7 days).

🔒
Critical for GDPR right-to-be-forgotten: delete a user's PII with DELETE, then after your retention window run VACUUM to physically remove the underlying file. Time travel to before the delete also becomes unavailable post-VACUUM.

Explain the Small File Problem and how Auto-Compaction in Databricks solves it.
Performance
▼

Streaming jobs and frequent small batch writes create thousands of tiny Parquet files — each requiring a metadata call to open, with Spark creating one task per file causing extreme scheduler overhead.

Auto-Compaction (Databricks only): After each write to a Delta table, Databricks checks if there are too many small files. If so, it automatically runs a compaction job in the background merging them into larger files (~128MB). This happens asynchronously without blocking your write.

ALTER TABLE my_table SET TBLPROPERTIES (
  'delta.autoOptimize.autoCompact' = 'true',
  'delta.autoOptimize.optimizeWrite' = 'true'
)

Optimized Write (companion feature): Instead of writing one file per partition per task, Databricks shuffles data before writing so each output file is ~128MB — reducing the file count from the write itself.

Explain the Data Skipping mechanism in Delta Lake. How does it differ from traditional SQL indexing?
Optimization
▼

When you read from a Delta table with a filter (e.g., WHERE event_date = '2024-01-15'), Delta reads the transaction log's file-level statistics before scanning any data. For each Parquet file, the log stores: min value, max value, and null count per column.

If a file's event_date range is 2024-01-10 to 2024-01-12, Delta knows it cannot possibly contain 2024-01-15 and skips it entirely — no I/O, no task created.

vs. traditional SQL index: A B-tree index points to exact row locations — ideal for OLTP single-row lookups. Delta's data skipping is coarser (file-level, not row-level) but is designed for analytical scans across billions of rows in object storage where random I/O is impractical.

💡
Z-Ordering colocates related data within files: OPTIMIZE table ZORDER BY (customer_id, event_date). This makes min/max statistics tight and effective — maximizing data skipping hits on those columns.

How do you choose between Partitioning and Z-Ordering for a 10TB dataset?
Architecture Decision
▼

These solve different problems and work best together:

Partitioning creates physical directory splits. A query with WHERE year=2024 reads only that folder — extremely effective for the partition column. Problem: high-cardinality columns (user_id with millions of values) → millions of tiny directories.

Z-Ordering works within partitions. It reorders rows so rows sharing values in Z-ordered columns are stored together in the same files, making data skipping (min/max) effective for those columns.

  • Pick 1-2 low-cardinality columns that appear in almost every query filter → Partition by these (e.g., date, region).
  • Pick 2-3 higher-cardinality columns that appear in many filters but aren't suitable for partitioning → Z-Order by these (e.g., customer_id, product_id).
  • Never partition by a high-cardinality column — this creates the severe small files problem.

How would you optimize a slow SQL query? What steps and tools do you use?
Performance
▼
  1. Read the execution plan first: EXPLAIN ANALYZE SELECT ... — look for full table scans, sort operations, nested loops, and large gaps between estimated and actual row counts.
  2. Check indexes: Is there an index on the filter columns (WHERE, JOIN ON)? Are existing indexes being used or bypassed by the optimizer?
  3. Reduce the dataset early: Push WHERE filters as early as possible. Avoid SELECT * — select only the columns needed.
  4. Avoid functions on indexed columns: WHERE YEAR(created_at) = 2024 prevents index use. Rewrite as: WHERE created_at >= '2024-01-01' AND created_at < '2025-01-01'
  5. Check join order and type: Join smaller tables first. Ensure join columns are indexed on both sides.
  6. Update statistics: UPDATE STATISTICS (SQL Server) or ANALYZE (PostgreSQL). Stale stats produce suboptimal execution plans.
  7. Avoid correlated subqueries: Replace them with JOINs or CTEs — correlated subqueries execute once per row in the outer query.
  8. In Synapse: Check distribution skew with DBCC PDW_SHOWSPACEUSED, update statistics, consider BROADCAST hints on small dimension tables.
2
Explain database indexing and its impact on performance.
Fundamentals
▼

An index is a separate data structure (usually a B-tree) that stores a sorted copy of one or more columns along with pointers to the original rows. Instead of scanning every row, the database finds matching rows in O(log n) time.

  • Clustered index: The table's physical row order matches the index. One per table. Best for range queries on the clustered key.
  • Non-clustered index: A separate structure pointing into the table. Multiple allowed. Best for frequently-filtered non-PK columns.
  • Covering index: Includes all columns needed by a query (using INCLUDE). The query is satisfied from the index alone — no table lookup required.
  • Columnstore index: Stores data column by column, highly compressed. Ideal for analytical aggregations over millions of rows.
⚖️
Indexes speed up reads but slow down writes — the index must be maintained on every INSERT, UPDATE, and DELETE. Index only columns that appear in WHERE, JOIN ON, and ORDER BY clauses of slow queries.

Explain the MERGE statement. How is it different from UPSERT?
Code
▼

UPSERT is a concept: insert if a record doesn't exist, update if it does. MERGE is the SQL statement that implements this — and more. MERGE is a superset of UPSERT — it can handle UPDATE, INSERT, and DELETE in one atomic statement.

MERGE INTO target_table AS t
USING source_table AS s
ON t.order_id = s.order_id
WHEN MATCHED AND s.amount != t.amount THEN
    UPDATE SET t.amount = s.amount, t.updated_at = GETDATE()
WHEN NOT MATCHED BY TARGET THEN
    INSERT (order_id, amount, created_at)
    VALUES (s.order_id, s.amount, GETDATE())
WHEN NOT MATCHED BY SOURCE THEN
    DELETE;  -- rows in target not in source → delete

The WHEN NOT MATCHED BY SOURCE THEN DELETE clause goes beyond UPSERT — it's a full sync that removes rows from target that no longer exist in source. MERGE is atomic — either all three operations succeed or none do.

    SELECT DISTINCT user_id, CAST(login_time AS DATE) AS login_date
    FROM user_logins
),

    FROM deduped

How do you manage secrets securely in a production pipeline?
Best Practice
▼
  • Never hardcode secrets in notebooks, config files, or pipeline definitions — even temporarily. They end up in version control and are extremely difficult to fully remove from git history.
  • Key Vault for all secrets. Rotate secrets in Key Vault without touching your code — services fetch the latest value on the next run automatically.

  • Environment-specific secrets: Separate Key Vaults (or secret versions) per environment — dev, staging, prod. Pipelines reference the environment-appropriate vault via a parameter.
  • Audit Key Vault access: Enable diagnostic logs → Log Analytics. Alert on unexpected access patterns or access from unexpected principals.
  • Least privilege: Grant each service identity only the specific secrets it needs — not blanket "Get All Secrets" on the entire vault.

How do you implement Column-Level Security and Row-Level Security in a Lakehouse?
Security
▼

Column-Level Security (CLS): Prevent specific users or roles from reading specific columns entirely.

-- Synapse Dedicated Pool
DENY SELECT ON dbo.customers(ssn, credit_card) TO analyst_role;
GRANT SELECT ON dbo.customers(customer_id, name, email) TO analyst_role;

-- Databricks Unity Catalog
GRANT SELECT ON TABLE customers TO analyst_role;

Row-Level Security (RLS): Restrict which rows a user can see based on their identity. In Synapse, create a security predicate function that evaluates the current user's access context, then bind it to the table as a SECURITY POLICY with FILTER PREDICATE — every query on the table automatically runs through the predicate.

In Databricks Unity Catalog: Define a column masking function and attach it to a column. Any query on that column runs through the masking function based on the caller's group membership — fine-grained, centrally managed access control across all tools querying through Unity Catalog.

How do you handle Schema Evolution without breaking downstream Power BI reports?
Architecture
▼

Core principle: insulate consumers from upstream schema changes using a stable interface layer.

  • Views as a contract layer: Power BI connects to a view (Synapse serverless view or Databricks SQL view), not directly to the Delta table. When upstream schema changes, update the view to maintain the same output columns — Power BI sees no change.
  • Semantic model isolation: In Power BI, use a semantic model (dataset) as the interface. Reports connect to the model, not the underlying source directly. Update the model's queries when schema changes, without touching any reports.
  • Additive changes are safe: Adding new columns to a Delta table doesn't break existing queries. Treat column removal or renaming as breaking changes requiring a deprecation period.
  • Column aliasing: If a source column is renamed, add an alias in the view to maintain backward compatibility during the transition period.
  • Schema change alerts: Add a validation step in your pipeline that compares the incoming schema to the expected schema and sends an alert before writing — giving you time to update the interface layer before new schema lands in production.

4
What are window functions in SQL and when would you use them?
ConceptWindow Functions
▼

A window function performs a calculation across a set of rows related to the current row — without collapsing them into a single output row like GROUP BY does. The "window" is defined by OVER (PARTITION BY ... ORDER BY ...).

Use window functions when you need both the individual row AND an aggregation — e.g., "show each employee's salary alongside their department average."

  • Ranking: ROW_NUMBER(), RANK(), DENSE_RANK() — rank rows within a group.
  • Running totals / moving averages: SUM() OVER, AVG() OVER with ORDER BY.
  • Lead/Lag: LAG(col, 1) — previous row's value; LEAD(col, 1) — next row's value. Great for month-over-month comparisons.
  • NTILE(n): Divide rows into n equal buckets (e.g., quartiles, deciles).
  • FIRST_VALUE / LAST_VALUE: First or last value in the window frame.
🗄 SQL — Window Function Examples
SELECT
  employee_id,
  department,
  salary,
  -- Rank within department
  RANK() OVER (PARTITION BY department ORDER BY salary DESC) AS dept_rank,
  -- Running total per department
  SUM(salary) OVER (PARTITION BY department ORDER BY employee_id
                  ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS running_total,
  -- Department average (no GROUP BY needed)
  AVG(salary) OVER (PARTITION BY department) AS dept_avg,
  -- Previous employee's salary
  LAG(salary, 1) OVER (PARTITION BY department ORDER BY salary DESC) AS prev_salary
FROM employees;

Explain normalization vs denormalization — when would you use each in a data engineering context?
DB Design
▼

Normalization eliminates redundancy by splitting data into multiple related tables (1NF → 2NF → 3NF). Each piece of information is stored exactly once. Benefits: consistent writes, smaller storage, no update anomalies. Best for OLTP systems (transactional databases — banking, e-commerce orders).

Denormalization intentionally introduces redundancy by merging tables or pre-computing aggregations. Benefits: fewer JOINs, faster reads, simpler queries. Best for OLAP systems (data warehouses, reporting) where read performance matters more than write efficiency.

  • Data Engineering rule of thumb: Normalize at the source/Bronze layer (preserve raw fidelity). Denormalize at the Gold layer (optimize for BI queries — flat wide tables, pre-joined dimensions).
  • Synapse Dedicated Pool: Replicated dimension tables (fully denormalized, no JOINs needed) + Hash-distributed fact tables = classic denormalized star schema.
  • Delta Lake: Use Delta tables at Silver (normalized, ACID) → Gold (wide denormalized tables optimized for reporting).
📌
In most modern data lakehouses: Normalize for correctness (Bronze/Silver) → Denormalize for performance (Gold/Serving layer).

A SQL query is running very slow on a large table. How would you troubleshoot and optimize it? You receive duplicate data from a source system — how do you handle it in your pipeline?
OptimizationPipeline Design
▼

Slow query — troubleshooting order:

  1. Run EXPLAIN / EXPLAIN ANALYZE — find the most expensive node (full table scan, sort, nested loop join).
  2. Check for missing indexes on filter and join columns.
  3. Check statistics are fresh — stale stats lead to bad execution plans. Run UPDATE STATISTICS or ANALYZE.
  4. Rewrite predicate — remove functions on indexed columns, use range conditions instead of BETWEEN on dates.
  5. Check JOIN order — filter the smallest set first before joining large tables.
  6. Add covering index if the query selects few columns from a heavily filtered table.
  7. Partition pruning — ensure the query's WHERE clause aligns with partition keys in Synapse or Delta.
  8. In Synapse: check for data skew with DBCC PDW_SHOWSPACEUSED, update statistics, add BROADCAST hints on small tables.

Handling duplicate data from source in a pipeline:

  • At ingestion (Bronze): Load raw data as-is — never deduplicate at Bronze. Log the duplicate count as an audit metric.
  • At Silver (cleansing): Deduplicate using ROW_NUMBER() or dropDuplicates() on natural keys. Define a clear "winner" rule (latest timestamp, highest sequence number).
  • Use Delta MERGE into Silver/Gold with WHEN MATCHED THEN UPDATE — naturally idempotent, handles re-delivered records.
  • Add a data quality check after dedup — assert uniqueness on the key column and alert if violation detected.
  • Alert source team — duplicate data from source is a data contract violation. Raise it so the root cause is fixed upstream.

-- COALESCE: return first non-null value
SELECT COALESCE(phone, mobile, 'No Contact') AS contact
FROM customers;

-- (avoid divide-by-zero)
SELECT revenue / NULLIF(units, 0) AS revenue_per_unit
FROM sales;

SELECT * FROM orders WHERE discount IS NULL;

SELECT
  COUNT(*)          AS total_rows,      -- includes NULLs
  COUNT(discount)   AS rows_with_disc,  -- excludes NULLs

  AVG(COALESCE(discount, 0)) AS avg_disc_with_zero
FROM orders;

-- Use IS NULL check after a LEFT JOIN to find unmatched
SELECT o.*, c.name
FROM orders o
LEFT JOIN customers c ON o.customer_id = c.customer_id
WHERE c.customer_id IS NULL;  -- orders with no customer!

from pyspark.sql.functions import (
  current_date, current_timestamp,
  date_trunc, add_months, date_add,
  datediff, months_between,
  year, month, dayofmonth, quarter,
  date_format, last_day,
  to_date, to_timestamp
)

df = df.withColumn("today", current_date())

# Truncate to month start
df.withColumn("month_start", date_trunc("month", col("order_date")))

# Add / subtract
df.withColumn("next_month",  add_months(col("order_date"),  1))
df.withColumn("prev_month",  add_months(col("order_date"), -1))
df.withColumn("week_ago",    date_add(col("order_date"),   -7))

# Differences
df.withColumn("days_diff", datediff(current_date(), col("order_date")))
df.withColumn("months_diff", months_between(current_date(), col("order_date")).cast("int"))

# Extract parts
df.withColumn("yr",  year(col("order_date")))
df.withColumn("mth", month(col("order_date")))
df.withColumn("day", dayofmonth(col("order_date")))
df.withColumn("qtr", quarter(col("order_date")))

# Format
df.withColumn("fmt_date", date_format(col("order_date"), "yyyy-MM-dd"))

# Parse strings to date
df.withColumn("parsed_date", to_date(col("date_str"), "dd/MM/yyyy"))

# Last day of month
df.withColumn("month_end", last_day(col("order_date")))
